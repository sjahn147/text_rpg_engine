## 사용자 선호 프로세스 특징 분석

### 1. 문서 관리 철학

**특징**:
- ✅ **중요 문서를 계속 업데이트하는 방식** (새 문서 생성 최소화)
- ✅ **CHANGELOG를 항상 최신화** (단일 소스)
- ✅ **Deprecated 문서 정리** (아카이브로 이동)
- ✅ **문서를 지저분하게 늘리지 않음**

**근거**:
- `docs/onboarding/README.md`: "문서를 지저분하게 늘리지 말고 중요 문서를 계속 업데이트하는 방식"
- `docs/changelog/CHANGELOG.md`: "모든 주요 변경사항을 기록하는 단일 소스"
- Deprecated 문서 정리 작업 수행

### 2. 자동화 선호

**특징**:
- ✅ **자동화된 프로세스 선호** (수동 작업 최소화)
- ✅ **체계적인 관리** (일관된 방식)
- ✅ **에이전트 기반 자동화** (에이전트가 자동으로 작업)

**근거**:
- 프로젝트 관리 자동화 제안 요청
- Jira 스타일 자동화 시스템 요청
- 에이전트가 자동으로 작업할 수 있는 도구 요청

### 3. 상태 기반 관리

**특징**:
- ✅ **명확한 상태 정의** (ideation → epic → task → development → qa → audit → done → deprecated)
- ✅ **상태 추적** (어떤 작업이 어느 단계에 있는지 파악)
- ✅ **자동 전환** (조건 만족 시 자동으로 다음 단계)

**근거**:
- IMPLEMENTATION_STATUS.md에서 상태별 분류
- 워크플로우 기반 프로세스 제안

### 4. 비판적 검토

**특징**:
- ✅ **비판적 성과 평가** (낙관적 해석 지양)
- ✅ **실제 동작 중심 평가** (이론적 가능성보다 실제 테스트 결과)
- ✅ **문제점 솔직하게 기록** (은폐하지 않음)

**근거**:
- `docs/rules/코딩 컨벤션 및 품질 가이드.md`: "비판적 성과 평가 우선 개발"
- 실제 구현 상태 확인 작업 수행

### 5. 데이터 중심 개발

**특징**:
- ✅ **데이터베이스 스키마 우선** (코드보다 DB 설계 먼저)
- ✅ **데이터 무결성 중시** (SSOT, 트랜잭션)
- ✅ **타입 안전성** (Pydantic, 타입 힌트)

**근거**:
- `docs/rules/코딩 컨벤션 및 품질 가이드.md`: "데이터 중심 개발"
- 데이터 무결성 개선 작업 수행

### 6. TDD 기반 개발

**특징**:
- ✅ **테스트 주도 개발** (테스트 먼저 작성)
- ✅ **시나리오 기반 테스트** (단위 테스트뿐만 아니라)
- ✅ **품질 게이트** (테스트 통과, 커버리지 기준)

**근거**:
- `docs/ideation/code_qa/02_WORKFLOW_TDD.md`: TDD 워크플로우 정의
- QA 테스트 시스템 구축

### 7. 시각적 관리 도구 선호

**특징**:
- ✅ **대시보드 기반 관리** (Streamlit 대시보드 요청)
- ✅ **시각적 상태 표시** (칸반 보드 스타일)
- ✅ **인터랙티브 관리** (드래그 앤 드롭, 버튼)

**근거**:
- Streamlit 대시보드 요청
- 시각적 프로젝트 관리 선호

## 현재 정의된 프로세스 분석

### 1. TDD 워크플로우 (`docs/ideation/code_qa/02_WORKFLOW_TDD.md`)

**정의된 프로세스**:
```
Requirement → Test (Red) → Implement (Green) → Refactor → Quality Gate
```

**특징**:
- ✅ 테스트 주도 개발
- ✅ 시나리오 기반 테스트
- ✅ 품질 게이트 (커버리지 ≥ 80%, 린터 오류 없음)

**부족한 점**:
- ❌ 프로젝트 관리 통합 없음
- ❌ 상태 추적 없음
- ❌ 문서 자동화 없음

### 2. 자동화된 프로젝트 관리 제안 (`AUTOMATED_PROJECT_MANAGEMENT_PROPOSAL.md`)

**정의된 프로세스**:
```
ideation → epic → task → development → qa → audit → done → deprecated
```

**특징**:
- ✅ 상태 기반 워크플로우
- ✅ 자동 전환
- ✅ 문서 자동화

**부족한 점**:
- ❌ TDD 워크플로우와 통합 없음
- ❌ 대시보드 통합 없음
- ❌ YAML 제출 시스템 없음

### 3. 문서 관리 정책 (`docs/onboarding/README.md`)

**정의된 정책**:
- 중요 문서를 계속 업데이트
- CHANGELOG 항상 최신화
- Deprecated 문서 정리

**특징**:
- ✅ 문서 정리 철학 명확
- ✅ 단일 소스 원칙

**부족한 점**:
- ❌ 자동화 메커니즘 없음
- ❌ 프로세스 강제 없음

## 일치도 평가

### 잘 반영된 부분 ✅

1. **상태 기반 관리**: ✅ 잘 반영됨
   - 워크플로우에 명확한 상태 정의
   - 상태 전환 규칙 정의

2. **자동화 선호**: ✅ 부분 반영됨
   - 자동 전환 규칙 정의
   - 문서 자동화 계획
   - ⚠️ 실제 구현은 미완료

3. **문서 관리 철학**: ⚠️ 부분 반영됨
   - Deprecated 처리 계획
   - CHANGELOG 자동 업데이트 계획
   - ⚠️ 실제 자동화는 미완료

### 부족한 부분 ❌

1. **TDD 통합**: ❌ 통합되지 않음
   - TDD 워크플로우와 프로젝트 관리 워크플로우가 분리됨
   - development 단계에 TDD 프로세스 통합 필요

2. **대시보드 통합**: ❌ 없음
   - Streamlit 대시보드 설계는 있지만 통합되지 않음
   - 시각적 관리 도구 부재

3. **YAML 제출 시스템**: ❌ 없음
   - 에이전트 제출 시스템 설계는 있지만 구현 계획만 있음

4. **비판적 검토 프로세스**: ⚠️ 부분 반영됨
   - Audit 단계는 있지만 비판적 평가 기준이 명확하지 않음
   - 실제 동작 중심 평가 프로세스 부재

5. **데이터 중심 개발 통합**: ⚠️ 부분 반영됨
   - 프로젝트 관리 프로세스에 DB 스키마 우선 원칙 반영 필요

## 개선 제안

### 1. TDD 워크플로우 통합

**현재 문제**:
- TDD 워크플로우와 프로젝트 관리 워크플로우가 분리됨
- development 단계에 TDD 프로세스가 명시되지 않음

**개선 방안**:
```
development 단계를 세분화:
  development
    ├── requirement (요구사항 정의)
    ├── test_red (테스트 작성 - Red)
    ├── implement_green (구현 - Green)
    ├── refactor (리팩토링)
    └── quality_gate (품질 게이트)
```

**통합된 워크플로우**:
```
ideation → epic → task → 
  development (TDD):
    requirement → test_red → implement_green → refactor → quality_gate →
  qa → audit → done → deprecated
```

### 2. Integrity 감사 프로세스 강화

**현재 문제**:
- Audit 단계의 Integrity 검증 기준이 명확하지 않음
- 핵심 규칙 준수 여부 체크 프로세스 부재

**개선 방안**:
- **Audit 단계에 Integrity 체크리스트 추가**:
  ```yaml
  integrity_checks:
    three_layer_architecture: passed|failed  # 3계층 구조 준수
    uuid_compliance: passed|failed  # UUID 규칙 준수
    data_centric_compliance: passed|failed  # 데이터 중심 개발 준수
    type_safety_compliance: passed|failed  # 타입 안전성 준수
    async_first_compliance: passed|failed  # 비동기 우선 개발 준수
    transaction_compliance: passed|failed  # 트랜잭션 규칙 준수
    migration_compliance: passed|failed  # 마이그레이션 규칙 준수
  ```

- **자동화된 Integrity 체크**:
  - 정적 분석 도구로 아키텍처 위반 탐지
  - UUID 사용 패턴 검증
  - 타입 힌트 누락 검증
  - 비동기 규칙 위반 탐지

### 3. 데이터 중심 개발 통합

**현재 문제**:
- 프로젝트 관리 프로세스에 DB 스키마 우선 원칙이 반영되지 않음

**개선 방안**:
- **requirement 단계에 DB 스키마 검토 추가**:
  ```
  requirement:
    - 기능 요구사항 정의
    - DB 스키마 검토 (필요 시 스키마 변경)
    - 데이터 무결성 검토
  ```

### 4. Streamlit 대시보드 통합

**현재 문제**:
- 대시보드 설계는 있지만 프로세스에 통합되지 않음

**개선 방안**:
- **모든 상태 변경을 대시보드를 통해 수행**:
  - 에이전트는 YAML로 제출
  - 사용자는 대시보드에서 승인/거부
  - 상태 변경 시 자동으로 문서 이동

### 5. YAML 제출 시스템 통합

**현재 문제**:
- YAML 제출 시스템이 프로세스에 명시되지 않음

**개선 방안**:
- **에이전트 작업 프로세스에 YAML 제출 필수화**:
  ```
  에이전트 작업:
    1. 코드 구현
    2. TODO YAML 제출 (docs/project-management/submissions/{TODO_ID}.yaml)
    3. 상태 전환 요청 YAML 제출
    4. 대시보드에서 승인 대기
  ```

### 6. 문서 자동화 강화

**현재 문제**:
- 문서 자동화 계획은 있지만 실제 프로세스에 통합되지 않음

**개선 방안**:
- **상태 변경 시 자동 문서 처리**:
  - 문서 이동
  - 메타데이터 업데이트
  - CHANGELOG 업데이트
  - IMPLEMENTATION_STATUS.md 업데이트
  - 관련 문서 링크 업데이트

## 통합된 프로세스 설계

### 통합 워크플로우

```
┌─────────────────────────────────────────────────────────┐
│  사용자 요청                                             │
└────────────────────┬────────────────────────────────────┘
                     ↓
┌─────────────────────────────────────────────────────────┐
│  Ideation (아이디어)                                    │
│  - ideation 문서 생성                                   │
│  - Streamlit 대시보드에 표시                            │
└────────────────────┬────────────────────────────────────┘
                     ↓ (승인)
┌─────────────────────────────────────────────────────────┐
│  Epic (대규모 기능)                                     │
│  - Epic 문서 생성                                        │
│  - Streamlit 대시보드에 표시                            │
└────────────────────┬────────────────────────────────────┘
                     ↓ (승인)
┌─────────────────────────────────────────────────────────┐
│  Task (작업 단위)                                       │
│  - Task 문서 생성                                        │
│  - Streamlit 대시보드에 표시                            │
└────────────────────┬────────────────────────────────────┘
                     ↓ (승인)
┌─────────────────────────────────────────────────────────┐
│  Development (TDD 프로세스)                             │
│  ├── Requirement                                        │
│  │   - 요구사항 정의                                     │
│  │   - DB 스키마 검토 (필요 시)                         │
│  ├── Test (Red)                                        │
│  │   - 실패하는 테스트 작성                              │
│  │   - 시나리오 테스트 포함                              │
│  ├── Implement (Green)                                 │
│  │   - 최소 구현                                        │
│  │   - TODO YAML 제출 (에이전트)                        │
│  ├── Refactor                                          │
│  │   - 코드 개선                                        │
│  │   - 리팩토링 YAML 제출 (에이전트)                     │
│  └── Quality Gate                                      │
│      - 테스트 통과 확인                                 │
│      - 커버리지 ≥ 80% 확인                             │
│      - 린터 오류 없음 확인                              │
│      - Quality Gate YAML 제출 (에이전트)                 │
└────────────────────┬────────────────────────────────────┘
                     ↓ (Quality Gate 통과)
┌─────────────────────────────────────────────────────────┐
│  QA (품질 보증)                                         │
│  - QA 테스트 실행                                       │
│  - QA YAML 제출 (에이전트)                              │
│  - Streamlit 대시보드에서 승인 대기                     │
└────────────────────┬────────────────────────────────────┘
                     ↓ (QA 통과)
┌─────────────────────────────────────────────────────────┐
│  Audit (비판적 검토)                                    │
│  - 실제 동작 검증                                        │
│  - 시나리오 테스트 실행                                  │
│  - 문제점 솔직하게 기록                                  │
│  - Audit YAML 제출 (에이전트)                           │
│  - Streamlit 대시보드에서 승인 대기                     │
└────────────────────┬────────────────────────────────────┘
                     ↓ (Audit 승인)
┌─────────────────────────────────────────────────────────┐
│  Done (완료)                                           │
│  - CHANGELOG 자동 업데이트                              │
│  - IMPLEMENTATION_STATUS.md 자동 업데이트               │
│  - 관련 문서 상태 업데이트                               │
│  - Streamlit 대시보드에 완료 표시                       │
└────────────────────┬────────────────────────────────────┘
                     ↓ (모든 관련 작업 완료)
┌─────────────────────────────────────────────────────────┐
│  Deprecated (폐기)                                      │
│  - ideation 문서 아카이브                               │
│  - [deprecated] 접두어 추가                             │
│  - CHANGELOG에 deprecation 기록                         │
└─────────────────────────────────────────────────────────┘
```

### Streamlit 대시보드 통합

**대시보드 역할**:
1. **상태 시각화**: 모든 작업의 현재 상태 표시
2. **상태 변경**: 드래그 앤 드롭 또는 버튼으로 상태 변경
3. **에이전트 제출 승인**: YAML 제출 승인/거부
4. **자동 문서 관리**: 상태 변경 시 자동으로 문서 이동

**에이전트 작업 흐름**:
1. 코드 구현
2. YAML 제출 (`docs/project-management/submissions/{TODO_ID}.yaml`)
3. 대시보드에 제출 알림
4. 사용자 승인 대기
5. 승인 시 자동 처리 (문서 이동, CHANGELOG 업데이트 등)

### YAML 제출 시스템

**제출 형식**:
- TODO 제출: `docs/project-management/submissions/TODO-{ID}.yaml`
- 상태 전환 요청: `docs/project-management/submissions/TRANSITION-{ID}.yaml`
- 진행 상황 업데이트: `docs/project-management/submissions/PROGRESS-{ID}.yaml`

**처리 프로세스**:
1. YAML 파일 감지
2. 유효성 검증
3. Streamlit 대시보드에 표시
4. 사용자 승인/거부
5. 승인 시 자동 처리
